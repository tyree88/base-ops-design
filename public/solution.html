Step 0 · today, up close

Every team has good tools. The glue between them is a person.

At 500+ events a month, each tool still does its one job well. What breaks is the connective tissue between them: someone copying status between systems, chasing confirmations, re-keying the same fact in four places. The hand-work itself, not the people doing it, is where error and delay creep in.

The same trap caught the apps that promised to fix it. Their bottleneck was never organizing information. It was capturing it without stopping the work that produces it. So the answer isn't a better place to file things; it's to remove the manual capture entirely. This is the picture after the design is built, not a restatement of the problems.

The fix is structural — not more tools

One record, gated stages, one owner each

It starts with one atom, the Event: the single unit the whole operation is measured in, the thing every date, venue, budget, and owner attaches to. Get that one unit right and three structural moves carry the rest: process and data design, done first. Automation is the test of a clean substrate, not a substitute for one.

One official record per event
Every system points at the same Event by one stable ID — one permanent label, identical in every tool (think Event #1042) — and holds a view, never its own copy. This dissolves "one venue, five spellings, five trackers" at the data layer.
Gated handoffs
An event can't move to the next stage until the named information that stage needs exists and a named owner accepts it. Missing or late facts can't propagate silently.
One accountable owner per stage
Every stage has exactly one role that owns it and drives it to the next gate: never shared, never orphaned.

Each system references the same Event by its stable ID: one atom, many views, never a second copy. Each holds a pointer to that one record.

The principle

Keep the smallest set of true facts in front of whoever acts next

Those three moves share one discipline: keeping the smallest set of true, current facts in front of whoever acts next, instead of piling on more tools or more copies. A capable operator, a person or a machine, works best on a tight, authoritative set of facts and worst when it has to reconcile five conflicting copies first. The five competing trackers aren't a tooling gap; they're noise crowding out signal. (The emerging name for this discipline is context engineering.)

Seen this way, the two things actually missing are the two things you engineer:

A captured memory A durable, versioned record that lives outside any one person's head or inbox, so the organization stops re-learning itself every cycle.
A living doctrine The "how" (the SOPs) held at the right altitude: standardized structure with strong, reusable heuristics rather than brittle rules or vague guidance.

Memory and doctrine are the connective tissue, made durable.

The operating layer

The machine does the curating

Keeping that smallest set of facts current is the job — and it's the one piece a person shouldn't do by hand. Today, the glue is a person. In the future state, capture is automatic: an operating layer reads each system on a schedule, pulls only what changed, and writes it once to the Event's record. People stop re-entering; the system does the glue, and the effort once spent on hand-work is reclaimed for the judgment, relationships, and exceptions only people can handle.

Base, instantiated

Every event, sourced to settled, on one path

The principle, made concrete for Base: every event becomes the core unit the whole machine turns on: sourced, confirmed, run, and settled through one clean path that mostly runs itself, so operating cost grows slower than the number of events. (Illustrative target — to validate in discovery.)

One record, zero reconciliation
Every fact about an event lives in one place. Nothing to reconcile across tools, nothing dropped between them. What you read is what's true.
The system does the chasing
Confirmations and receipts are followed up automatically. People stop hand-chasing status; the routine work simply happens on time.
A live map, not a doc
The whole operation is navigable as a living map: every event, owner, and connection in one view, so anyone can see where things stand at a glance.
Each event advances itself
An event moves through clean, owned stages on its own. Each step has one accountable owner and can't advance until it's truly ready.
What this earns
  • Scale without scaling the team — the same team runs 500+ events a month across dozens of cities, because the busywork per event falls, not because anyone is replaced.
  • Every event confirmed in time — the soft-hold-to-signed gap closes, so events stop falling through at the last minute.
  • Nothing drops silently — when something slips, it surfaces as a flagged exception, never a quiet failure.
  • A new city goes live on day one — it inherits the whole system instead of reinventing one.
