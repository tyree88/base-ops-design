Operational framework for Base

An ops system for a membership social club scaling from ~20 to 500+ events per month across dozens of cities. The brief grades reasoning, not tools.

The big idea. The seven problems Base listed boil down to two root causes: there is no single official record of an event that someone owns, and no clear owner for the process around it. So the real gap is a missing foundation (one official record of each event, plus a step-by-step process where every stage has a named owner), not missing tools. Everything Base wants (confirmations that hold, a view everyone trusts, automation later) only works once that foundation exists.

Problem & frame

How the problem was understood before any solution was drawn.

The brief in one line. Build an operational framework for dozens of cities and hundreds of events a month, minimizing manual intervention and keeping reliability. The load-bearing sentence: "automation alone will not solve these problems if the underlying process design is flawed."

Current stack: assume it exists, but don't assume it's optimal

Asana (tasks) · Slack (team chat) · Missive (shared inboxes / external comms) · Airtable (operational + city data). The problem isn't any single tool. It's that four systems hold four partial copies of an event's truth and none is authoritative. A venue change lands in a Missive inbox and lives there; that's where "plan for 80, room holds 50" is born.

Seven challenges → two root causes + symptoms — Decision 1, sort before you solve (D1)

LayerItems
Root — data/structure#1 information fragmented across systems/owners · #2 event vs operational data siloed
Root — ownership/knowledge#3 process ownership unclear · #7 knowledge trapped in individuals
Symptoms#4 manual follow-ups · #5 missing/late info → downstream failures · #6 undefined escalation

Fix the two roots and the symptoms dissolve. Sort before solving, rather than mopping the floor with the tap running.

The frame, step by step

0
Don't anchor on the tools. "What's missing?" means nothing against a list of tools, only against a model of the functions that have to exist.
1
Don't accept the brief's list as-is. Are the 7 problems equals, or do some cause the others? That turns a to-do list into a sort.
2
Root-cause vs symptom test. "If every other problem were fixed, would this one still exist?" #5/#4/#6 vanish → symptoms; #1/#2/#3/#7 stand alone → root causes.
3
The hinge — test the tools framing. "If every tool were perfectly integrated tomorrow, would the problem be solved?" No — there's still no agreement on what the one official record is or who owns it. The problem flips from tooling to foundation.
4
Restate it positively. Base has no operating system for its core unit of work — one official, owned record per event + a defined process around it.
5
Set the boundary — Decision 3, model the operations side (D3). Scope = the venue and operations side (from sourcing an event to settling its costs). The demand side (member acquisition, matching) is out; it only touches this at event sizing. has a gap the wording later leaks member money back in.

Scale is several dimensions, not one number

20→500 events/month is the headline, but the dimensions that actually break things differ: cities · venues per city · events per venue · ops headcount · hosts · same-night events running at once · geographic/timezone spread. Each one stresses a different part of the system first.

Decisions

All 23 logged decisions — the choice and the why. Filter by how each held up under hard questioning, or search.

(D1)Sort before you solvesolid
7 challenges → two root-cause families + symptoms.
Root causes breed symptoms; fix the two roots and several dissolve — don't mop with the tap on.
(D2)The core unit is the Eventsolid
The core unit is the Event; actions, messages, costs, exceptions all hang off it.
They can't stand alone — everything attaches to the event.
(D3)Scope = operations sidehas a gap
Operations side only; demand side (member acquisition, matching) out.
The brief is venue comms + city operations. The wording has a gap — member money re-enters the design.
(D4)The set of things we trackfollows from others
Event, Venue, City, Host, Venue-Comms Thread, Receipt, Incident — each with a stable ID.
Follows from looking at lifecycle, persistence, how-many-to-how-many, and ownership.
(D5)One event, many venuessolid
One-to-many; receipts/confirmations/incidents tracked per venue.
Tours span venues; a single event-level "confirmed" flag hides a venue that quietly fell through. A one-way door, so decide it now.
(D6)What a hand-off issolid
Hand-off = a stage change + a required information package + a named owner + a due date (gated).
Structurally stops missing or late info from spreading (paired with Decision 18, exceptions stay in the system (D18)).
(D7)One official recordsolid
Split by domain — one event record + one finance book + one playbook; chat is never the record.
"One tool for everything" is brittle; one authoritative home per domain is defensible.
(D8)Who owns whatsolid
Two dimensions — headquarters vs the local city team, and by lifecycle stage.
Standard pattern for a business with many local branches; gives unclear ownership a structural answer.
(D9)The event lifecyclesolid
Sourced → Drafted → Venue-Requested → Venue-Booked → Confirmed → Run → Reconciled → Closed (+ Cancelled, Rescheduled).
Booked (a soft hold) is deliberately not Confirmed (signed) — that's the seam where Base fails; Reconciled (money settled) is split from Closed (postmortem done).
(D10)One accountable owner per stagehas a gap
Exactly one accountable owner for each stage.
Kills "everyone assumed someone else had it." The gap: the Incident owner is a phantom "Central Ops" that isn't in the list of roles.
(D11)Multi-venue confirmationopen choice
Per-venue status rolls up; the event only advances when every required venue is ready; a venue dropping out after advancing raises an Incident, never a silent gap.
One event-level flag hides a venue that quietly fell through. Gap: what members are told on a partial failure.
(D12)Gated hand-offs + deadlineshas a gap
Every stage change needs a named information package before the next owner accepts it; deadlines anchored to the event date.
Puts Decision 6, what a hand-off is (D6) into practice. The gap: the deadline numbers are placeholders — flag them as illustrative or remove.
(D13)One status, four viewshas a gap
One official status, four views (city / finance / member / venue); members see it only once the venue is actually signed.
The public view can't get ahead of signed reality. The gap: no reverse gear for member money or visibility when an event is rescheduled or cancelled.
(D14)Decide and moveopen choice
Lock the recommended side of each open choice as version 1; each one is reversible in a line.
Don't stall the machine on easily reversible calls. Caveat: it needs a review gate — without one it let gaps ride unchecked.
(D15)Official record, finalizedsolid
Split by system role (operations record / finance book / playbook / coordination chat) + 4 rules; money = planned budget vs actual spend, joined at reconciliation.
Separates concerns that change at different rates and need different controls; the rule "a fact isn't real until it's written to the official record" kills the "plan for 80, room holds 50" failure.
(D16)Standardize structure, localize contentopen choice
Standardize the structure, localize the content — headquarters owns the data shape, templates, and procedures; the local city team owns the values, people, venues, and calendar.
Data stays comparable and easy to roll up, and a new city inherits the system on day one; local content still respects what fits each local venue.
(D17)Dependency mapfollows from others
Mapped across systems, teams, and processes, marking what breaks if each link snaps; a gate is a hardened link; two exposures remain.
Shows the architecture already hardens most links and names the two that aren't (knowledge between headquarters and the local city team; keeping the joined records in sync).
(D18)Exceptions stay in the systemsolid
An authorized override → opens an Incident + a required back-fill + a visible risk flag, never silent; hard limits for money going out, opening public sales, and closing the books clean.
A gate you can't override gets routed around; an authorized one keeps the exception inside the system and turns the override rate into a health metric.
(D19)Risk & scalehas a gap
Failure modes trace back to the 2 root causes; "scale" is several different dimensions; three of them need explicit mitigation.
Harden each dimension on its own. The gap: "cost grows slower than volume" depends on automation, not yet on structure; the same-night-events mitigation leans on the (still unsolved) knowledge loop.
(D20)Build orderfollows from others
Foundations first → process → data → standardization → automation; automation is the test of a clean process, not the foundation.
The brief's own thesis: automation won't fix a flawed process.
(D21)Migrationsolid
Run old and new side by side, city by city: new events start on the new lifecycle, in-flight events finish on the old; one pilot city first; a named owner reconciles the overlap; stable IDs are back-filled; governed by the override-and-back-fill path from Decision 18, exceptions stay in the system (D18).
No big-bang rebuild is possible while 200+ events a month run live — how to cut over is the first rollout decision. Added after the pressure-test.
(D22)Owner continuitysolid
Every stage names a successor + an escalation path to the City Manager, so an owner who vanishes or is unavailable never strands an event mid-stage.
One owner per stage gives accountability but is a single point of failure if that owner is lost; a successor rule restores continuity without splitting the accountable role. Added after the pressure-test; refines Decision 10, one accountable owner per stage (D10) and Decision 19, risk and scale (D19).
(D23)Playbook at scalesolid
The playbook-update loop carries its own deadline (past it → auto-escalate) and tiers to regional playbook owners as cities grow.
New lessons re-spike with each new market; an unbounded central-approval delay would break consistency exactly at scale. Added after the pressure-test; refines Decision 19, risk and scale (D19).

One official record & how information flows

RoleThe official record for
Operations record (the system of record)the event's operational state — status, dates, capacity, planned budget, ownership, venue links
Finance booksettled money — invoices, payments, reconciliation actuals
Playbookthe "how" — standard procedures, templates, data shape, approval thresholds
Coordination (chat)nothing — it's temporary; anything that matters gets written to the operations record

The four rules: ① one official home per fact · ② refer to things by their stable ID, never their name · ③ a fact isn't real until it's written to the official record · ④ read across domains by reference, not by copying. Rule ③ is the structural fix for the "plan for 80, room holds 50" failure.

One status, many views: at Venue-Booked finance sees a deposit at risk while members see only a tentative "save the date"; at Confirmed, sales open against a signed venue. The member view is read straight from the one status, so the public state can't get ahead of signed reality.

Decision 11, multi-venue confirmation (D11)  open choice / mostly solid

ChallengeDecision 5 (D5) committed you to one event, many venues. How does a three-venue tour reach "Confirmed" without the record lying?
Our answerTrack a sub-status per venue. Every required venue runs its own line — Targeted → Requested → Soft-Held → Signed → Run-Ready. The event's state rolls up from those lines; it's not a free-standing flag.
ChallengeOne slow venue freezes a finished event. At 500 a month that's a throughput killer. Defend the hold.
Our answerThe alternative is advancing while a venue is unsigned — exactly the case where a venue quietly fell through while the event still reads "confirmed." The release valve is the word required: only required venues gate; optional stops never hold the event.
ChallengeEvent's Confirmed, tickets out, and Wednesday a required venue cancels. What happens?
Our answerNever silent — the venue flips to Fallback-Pending, the event carries a venue-risk flag, and it raises an Incident. But the member holding a ticket to the dropped leg? …Not specified. A real gap.
Verdict — open choice / mostly solid. The roll-up and raising-an-Incident-on-drop defend; what members are told on a partial failure is still open.

Decision 12, gated hand-offs: information package + deadline (D12)  deadlines have a gap

ChallengeWhy does the information package matter more than the deadline?
Our answerThe package is the gate — the precise list of what must exist before the next owner accepts. No package, no advance. That's what makes missing info structurally unable to spread.
ChallengeYour deadlines — 7 days out, 21 days out, five business days. Where did those numbers come from?
Our answer…Placeholders. The defensible claim is the shape — late gates anchor to the event date, early gates run in business days — not the exact values. They wait on the "current baseline" discovery question (Q-base), which we never had.
ChallengeThen say so loudly or cut them — a reader can't tell a placeholder from a calibrated deadline.
Our answerAgreed. Flag them as illustrative or remove them. The package-gated mechanism stands regardless.
Verdict — solid on the mechanism, has a gap on the deadline values.

Decision 13, one status four views (D13)  has a gap · fix first

ChallengeWhy is "one record, many views" better than each team tracking its own status?
Our answerPrivate copies are how "five spellings, five trackers" is reborn. If the member view is read straight from the one status — not flipped by hand — the public state can never get ahead of signed reality.
ChallengeEvent is Confirmed, sales open, members hold tickets. It gets Rescheduled — back to Venue-Requested. What does the member view read there?
Our answer…Invisible. The state reverts below the visibility line, but the member obligations created at Confirmed don't revert with it. There's no row for "sold, now un-sold."
ChallengeSame for Cancel after sales — where's the refund of member money?
Our answerNot modeled. Cancel handles the venue side and sunk cost, but never returns members' money. The most financially active moment in the lifecycle has no reverse gear. Biggest correctness gap.
Verdict — has a gap. Status-read-straight-from-the-record holds; member money and visibility on reschedule + cancel is unmodeled. Highest-priority fix.

Decision 14, decide and move (D14)  open choice · method on trial

ChallengeDefend deciding by default — locking open choices as version 1 without live sign-off.
Our answerReversibility. Every open choice is a two-way door — one rule flips it. Stalling the machine on calls that reverse in a line buys no reasoning value.
ChallengeBut the gaps — Decision 10 (D10), Decision 12 (D12), Decision 13 (D13) — all rode in under this. The method removed the catch.
Our answerAccepted, with a correction: deciding by default is sound for reversible choices only with a mandatory review gate before calling it "done." The flaw was treating "locked version 1" as finished without the audit.
Verdict — open choice / solid, with a caveat. Fine for two-way doors, but only with a real review gate.

Decision 15, official record finalized (D15)  solid

Challenge"Split by domain" is a label — give me the mechanism that stops four silos.
Our answerFour rules: one domain per fact; refer to things by their stable ID; a fact isn't real until it's written to the official record; read across domains by reference, not by copying.
ChallengeSplitting the record means you have to join it back. Where do the domains meet, and what breaks?
Our answerThe money boundary: the operations record owns the planned budget and status, the finance book owns invoices and settlement; they meet at reconciliation. Between reconciliations, ongoing drift isn't watched — a named exposure, not solved.
Verdict — solid. Four rules + the reconciliation join defend cold; watching for drift between cycles is the one exposure.

Risks & gaps

What the deep review and the pressure-test exposed. The core held; six things need fixing before submission.

Top risk — defensibility. Most of Decisions 10–20 ((D10)–(D20)) and all three deliverables were generated by automated helper agents. In a reasoning-graded exercise, the first gap to close is whether every decision can be defended cold — the polish currently runs ahead of the practiced argument.

The six gaps

P0Phantom role — Incident ownership (Decision 10 (D10) / Decision 18 (D18))
The list of things we track names "Central Ops / on-call" as the Incident owner; it's not in the list of roles and contradicts Decision 18 (D18) (City Manager escalation). The exception system's owner is undefined. Fix: assign it to the City Manager + define an on-call tier; reconcile it across the things-we-track table, the who's-accountable grid, and Decision 18 (D18).
P0Reschedule/Cancel after member sales (Decision 9 (D9) / Decision 13 (D13))
Member RSVP opens at Confirmed, but Reschedule reverts the event to a state the member view renders invisible, and Cancel never models refunds of member money. Fix: a member-obligation sub-state that travels through a reschedule + a refund step inside cancel.
P1"Cost grows slower than volume" is oversold (Decision 19 (D19))
It's the headline success metric, but the gates require a human to accept each event — labor that grows in step with volume until the (deferred, unbuilt) automation layer. Fix: reframe as "reliability now; cost grows slower than volume after automation."
P1Same-night-events fix is circular (Decision 19 (D19))
Many same-night events are patched with "playbooks" — the knowledge loop, which is itself a flagged unsolved exposure. Fix: sequence the knowledge loop first, or define an interim manual escalation roster.
P1Scope wording is now false (Decision 3 (D3))
"Demand only touches this at sizing" — but member money is all over the design. Fix: widen the boundary statement to "demand touches this at sizing and at the money/comms tied to a specific event."
P2Deadline values are unfounded (Decision 12 (D12))
The 7-days-out / 21-days-out / 5-day numbers have no basis. Fix: flag them as illustrative pending the "current baseline" discovery question (Q-base), or remove.

Exception model (Decision 18 (D18)) — the win condition

An authorized role can force-advance past an unmet gate — but it opens an Incident (which gate, who, why) + a mandatory back-fill at reconciliation + a visible risk flag. Never a silent skip. A few gates are hard (you can't override them): money going out needs a signature, opening public sales needs a real signature, and closing needs a clean ledger. Because every override is an Incident, the override rate becomes a health metric — it tells you which gates are too rigid versus where operations keeps failing, feeding the playbook loop.
